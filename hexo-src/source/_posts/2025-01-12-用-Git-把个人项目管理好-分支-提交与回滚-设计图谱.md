> 把项目当成产品管理，分支策略就是你的时间轴。

<!-- more -->

这篇面向独立开发者，从成本视角拆解用 Git 把个人项目管理好，目标是用更少资源支撑同样吞吐。

我们先统一指标（QPS/成本、存储成本），再把动作拆成可验证步骤。

## 问题拆分

个人项目最容易丢失上下文：为什么改？改了什么？如何回退？如果提交记录足够清晰，你可以在任何时间重建决策链路。

当团队是独立开发者规模时，问题往往不是“不会做”，而是“难以复用”。

## 设计原则

- 目标清晰：围绕QPS/成本、存储成本建立一致口径
- 拆分步骤可验证、可回滚
- 权衡边界：避免过度优化导致投入失衡

## 分步实施

- 主干始终可发布，功能分支短周期完成。
- 提交信息用“动词 + 影响”描述改动目的。
- 发布点打 tag，回滚用 revert 保留可追溯历史。

## 关键细节

在执行用 Git 把个人项目管理好时，建议把日志和监控作为“第二视角”，及时发现偏差。

如果资源有限，优先把用更少资源支撑同样吞吐放在最靠近用户的路径上。

## 常见误区

- 长期堆积未合并的分支
- 提交信息没有语义
- 在公共分支上强推覆盖历史

## 下一步建议

- 把每周一发布固定为节奏约束，形成习惯
- 补齐工具链：文档：飞书/Notion 需求记录 + 监控：Grafana + Prometheus
- 为关键环节写一份可复用的检查清单

## Checklist

- 功能分支 < 3 天完成
- 合并前自测并备注风险点
- 关键改动写在 commit body
