## 开篇定位

这篇面向独立开发者，从性能视角拆解用 Git 把个人项目管理好，目标是缩短响应路径，降低等待时间。

深度定位：实战，重点是面向真实流量与团队协作。

我们先统一指标（P95/INP/TTFB），再把动作拆成可验证步骤。

## 场景画像

典型场景是「从 0 到 1 的新项目」与「独立开发者」并存：目标快速迭代，但资源有限。

我们从性能视角拆解 用 Git 把个人项目管理好，先锁定P95/INP/TTFB，再逐层拆解改动点。

## 目标与指标

- 目标：缩短响应路径，降低等待时间
- 衡量口径：P95/INP/TTFB
- 复盘节奏：每次发布后复盘一次

## 方案路径

- 主干始终可发布，功能分支短周期完成。
- 提交信息用“动词 + 影响”描述改动目的。
- 发布点打 tag，回滚用 revert 保留可追溯历史。

## 深度推进

- 容量评估与压测
- 流程标准化
- 持续优化与复盘

## 小型案例

下面的片段可作为最小闭环示例：

~~~bash
git switch -c feat/notes
# ... edit files
git add .
git commit -m "feat: add notes outline"
git tag v1.2.0
git revert <sha>
~~~

## 取舍与边界

- 如果只追求性能，可能带来新的风险：只优化局部导致瓶颈迁移
- 把复杂度锁在工具或中间层，避免侵入业务核心
- 优先保证可回滚，再谈极致优化
- 避免单点依赖
- 提前准备峰值与故障剧本

## 落地节奏

- 建议节奏：每周一发布
- 先小范围灰度，再扩大覆盖
- 每一步都要有可观察数据

## 执行提示

- 建立演练机制
- 对齐多角色协作
- 形成指标周报
- 深度目标：团队协作下稳定交付

## 工具清单

- 文档：飞书/Notion 需求记录

- 告警：Sentry/自定义报警
