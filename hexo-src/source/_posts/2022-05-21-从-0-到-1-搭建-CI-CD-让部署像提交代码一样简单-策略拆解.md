## 开篇定位

这篇面向多环境交付团队，从性能视角拆解从 0 到 1 搭建 CI/CD，目标是缩短响应路径，降低等待时间。

深度定位：进阶，重点是稳定可复用，开始处理边界。

我们先统一指标（P95/INP/TTFB），再把动作拆成可验证步骤。

## 场景画像

典型场景是「业务增长带来的容量压力」与「多环境交付团队」并存：目标快速迭代，但资源有限。

我们从性能视角拆解 从 0 到 1 搭建 CI/CD，先锁定P95/INP/TTFB，再逐层拆解改动点。

## 目标与指标

- 目标：缩短响应路径，降低等待时间
- 衡量口径：P95/INP/TTFB
- 复盘节奏：每次发布后复盘一次

## 方案路径

- 构建 → 单测 → 打包 → 产物入库 → 多环境发布。
- 把环境差异放进配置，避免人为操作。
- 发布保留回滚入口，失败时一键切换。

## 深度推进

- 补齐指标与监控
- 处理异常与回滚
- 形成可复用模板

## 小型案例

下面的片段可作为最小闭环示例：

~~~yaml
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm test
      - run: npm run build
~~~

## 取舍与边界

- 如果只追求性能，可能带来新的风险：只优化局部导致瓶颈迁移
- 把复杂度锁在工具或中间层，避免侵入业务核心
- 优先保证可回滚，再谈极致优化
- 稳定优先于新增能力
- 不要在无监控下扩范围

## 落地节奏

- 建议节奏：季度里程碑发布
- 先小范围灰度，再扩大覆盖
- 每一步都要有可观察数据

## 执行提示

- 逐步引入自动化检查
- 灰度发布并演练回滚
- 沉淀复盘文档
- 深度目标：方案可复用且可回滚

## 工具清单

- 协作：Jira/禅道
- 发布：灰度开关/特性开关
- 回滚：可切换版本包
