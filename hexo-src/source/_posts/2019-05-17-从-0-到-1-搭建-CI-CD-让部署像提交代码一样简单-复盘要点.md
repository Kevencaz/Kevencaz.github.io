> 部署应该是流水线，而不是手工仪式。

<!-- more -->

这篇面向多环境交付团队，从协作视角拆解从 0 到 1 搭建 CI/CD，目标是让协作可追踪、可回滚。

我们先统一指标（交付周期/回滚次数），再把动作拆成可验证步骤。

## 背景

把测试、构建、部署标准化后，团队可以把精力放在需求和质量上，而不是频繁救火。

在业务增长带来的容量压力阶段，最容易忽视的是规范缺失造成返工，先对齐度量口径能减少返工。

## 方法拆解

- 构建 → 单测 → 打包 → 产物入库 → 多环境发布。
- 把环境差异放进配置，避免人为操作。
- 发布保留回滚入口，失败时一键切换。

## 关键指标

- 核心指标：交付周期/回滚次数
- 目标：让协作可追踪、可回滚
- 验证方式：上线后 7 天回看趋势

## 示例

~~~yaml
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm test
      - run: npm run build
~~~

## 常见误区

- 流水线没有质量门禁
- 产物与配置混在一起
- 发布过程不可回滚
- 规范缺失造成返工

## 工具与资料

- 协作：Jira/禅道
- 发布：灰度开关/特性开关
- 回滚：可切换版本包

## Checklist

- 至少一条自动化测试
- 产物可复用且可追溯
- 每次发布都有回滚方案
- 节奏：季度里程碑发布
