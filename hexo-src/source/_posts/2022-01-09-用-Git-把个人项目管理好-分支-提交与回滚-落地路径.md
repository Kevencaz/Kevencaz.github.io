## 开篇定位

这篇面向独立开发者，从稳定性视角拆解用 Git 把个人项目管理好，目标是降低故障率并缩短恢复时间。

深度定位：进阶，重点是稳定可复用，开始处理边界。

我们先统一指标（SLA/MTTR/错误率），再把动作拆成可验证步骤。

## 场景画像

典型场景是「从 0 到 1 的新项目」与「独立开发者」并存：目标快速迭代，但资源有限。

我们从稳定性视角拆解 用 Git 把个人项目管理好，先锁定SLA/MTTR/错误率，再逐层拆解改动点。

## 目标与指标

- 目标：降低故障率并缩短恢复时间
- 衡量口径：SLA/MTTR/错误率
- 复盘节奏：每次发布后复盘一次

## 方案路径

- 主干始终可发布，功能分支短周期完成。
- 提交信息用“动词 + 影响”描述改动目的。
- 发布点打 tag，回滚用 revert 保留可追溯历史。

## 深度推进

- 补齐指标与监控
- 处理异常与回滚
- 形成可复用模板

## 小型案例

下面的片段可作为最小闭环示例：

~~~bash
git switch -c feat/notes
# ... edit files
git add .
git commit -m "feat: add notes outline"
git tag v1.2.0
git revert <sha>
~~~

## 取舍与边界

- 如果只追求稳定性，可能带来新的风险：缺少降级兜底引发雪崩
- 把复杂度锁在工具或中间层，避免侵入业务核心
- 优先保证可回滚，再谈极致优化
- 稳定优先于新增能力
- 不要在无监控下扩范围

## 落地节奏

- 建议节奏：每周一发布
- 先小范围灰度，再扩大覆盖
- 每一步都要有可观察数据

## 执行提示

- 逐步引入自动化检查
- 灰度发布并演练回滚
- 沉淀复盘文档
- 深度目标：方案可复用且可回滚

## 工具清单

- 文档：飞书/Notion 需求记录
- 监控：Grafana + Prometheus
- 告警：Sentry/自定义报警
