## 开篇定位

这篇面向独立开发者，从协作视角拆解用 Git 把个人项目管理好，目标是让协作可追踪、可回滚。

深度定位：进阶，重点是稳定可复用，开始处理边界。

我们先统一指标（交付周期/回滚次数），再把动作拆成可验证步骤。

## 背景

个人项目最容易丢失上下文：为什么改？改了什么？如何回退？如果提交记录足够清晰，你可以在任何时间重建决策链路。

在从 0 到 1 的新项目阶段，最容易忽视的是规范缺失造成返工，先对齐度量口径能减少返工。

## 方法拆解

- 主干始终可发布，功能分支短周期完成。
- 提交信息用“动词 + 影响”描述改动目的。
- 发布点打 tag，回滚用 revert 保留可追溯历史。

## 深度推进

- 补齐指标与监控
- 处理异常与回滚
- 形成可复用模板

## 关键指标

- 核心指标：交付周期/回滚次数
- 目标：让协作可追踪、可回滚
- 验证方式：上线后 7 天回看趋势

## 执行提示

- 逐步引入自动化检查
- 灰度发布并演练回滚
- 沉淀复盘文档

## 示例

~~~bash
git switch -c feat/notes
# ... edit files
git add .
git commit -m "feat: add notes outline"
git tag v1.2.0
git revert <sha>
~~~

## 常见误区

- 长期堆积未合并的分支
- 提交信息没有语义
- 在公共分支上强推覆盖历史
- 规范缺失造成返工
- 稳定优先于新增能力
- 不要在无监控下扩范围

## 工具与资料

- 文档：飞书/Notion 需求记录
- 监控：Grafana + Prometheus
- 告警：Sentry/自定义报警

## Checklist

- 功能分支 < 3 天完成
- 合并前自测并备注风险点
- 关键改动写在 commit body
- 深度目标：方案可复用且可回滚
- 节奏：每周一发布
