## 开篇定位

这篇面向多环境交付团队，从协作视角拆解用 Git 把个人项目管理好，目标是让协作可追踪、可回滚。

深度定位：入门，重点是建立基本概念与最小闭环。

我们先统一指标（交付周期/回滚次数），再把动作拆成可验证步骤。

## 背景

个人项目最容易丢失上下文：为什么改？改了什么？如何回退？如果提交记录足够清晰，你可以在任何时间重建决策链路。

在业务增长带来的容量压力阶段，最容易忽视的是规范缺失造成返工，先对齐度量口径能减少返工。

## 方法拆解

- 主干始终可发布，功能分支短周期完成。
- 提交信息用“动词 + 影响”描述改动目的。
- 发布点打 tag，回滚用 revert 保留可追溯历史。

## 深度推进

- 梳理核心术语
- 画出最小流程
- 先手工跑通一条路径

## 关键指标

- 核心指标：交付周期/回滚次数
- 目标：让协作可追踪、可回滚
- 验证方式：上线后 7 天回看趋势

## 执行提示

- 先解决“能用”再优化
- 小范围试点避免过度复杂
- 记录关键假设与结论

## 示例

~~~bash
git switch -c feat/notes
# ... edit files
git add .
git commit -m "feat: add notes outline"
git tag v1.2.0
git revert <sha>
~~~

## 常见误区

- 长期堆积未合并的分支
- 提交信息没有语义
- 在公共分支上强推覆盖历史
- 规范缺失造成返工
- 避免一次性全量替换
- 别在入门阶段追求极致优化

## 工具与资料

- 协作：Jira/禅道
- 发布：灰度开关/特性开关
- 回滚：可切换版本包

## Checklist

- 功能分支 < 3 天完成
- 合并前自测并备注风险点
- 关键改动写在 commit body
- 深度目标：能够独立完成最小落地
- 节奏：季度里程碑发布
