## 开篇定位

这篇面向2-5 人小团队，从成本视角拆解用 Git 把个人项目管理好，目标是用更少资源支撑同样吞吐。

深度定位：入门，重点是建立基本概念与最小闭环。

我们先统一指标（QPS/成本、存储成本），再把动作拆成可验证步骤。

## 背景

个人项目最容易丢失上下文：为什么改？改了什么？如何回退？如果提交记录足够清晰，你可以在任何时间重建决策链路。

在已有系统的重构期阶段，最容易忽视的是过度优化导致投入失衡，先对齐度量口径能减少返工。

## 方法拆解

- 主干始终可发布，功能分支短周期完成。
- 提交信息用“动词 + 影响”描述改动目的。
- 发布点打 tag，回滚用 revert 保留可追溯历史。

## 深度推进

- 梳理核心术语
- 画出最小流程
- 先手工跑通一条路径

## 关键指标

- 核心指标：QPS/成本、存储成本
- 目标：用更少资源支撑同样吞吐
- 验证方式：上线后 7 天回看趋势

## 执行提示

- 先解决“能用”再优化
- 小范围试点避免过度复杂
- 记录关键假设与结论

## 示例

~~~bash
git switch -c feat/notes
# ... edit files
git add .
git commit -m "feat: add notes outline"
git tag v1.2.0
git revert <sha>
~~~

## 常见误区

- 长期堆积未合并的分支
- 提交信息没有语义
- 在公共分支上强推覆盖历史
- 过度优化导致投入失衡
- 避免一次性全量替换
- 别在入门阶段追求极致优化

## 工具与资料

- 流水线：GitHub Actions
- 性能：Lighthouse/DevTools
- 质量：ESLint/单测

## Checklist

- 功能分支 < 3 天完成
- 合并前自测并备注风险点
- 关键改动写在 commit body
- 深度目标：能够独立完成最小落地
- 节奏：双周迭代 + 灰度 24 小时
